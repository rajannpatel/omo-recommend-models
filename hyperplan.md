# Dynamic Local Model Recommendation Engine Hyperplan

Provenance: generated by the `plan` agent after adversarial hyperplan team rounds for the `omo-recommend-models` CLI. User clarification: the old 1.5GB VRAM margin is no longer needed; use the requested fit budget `gpu.vramGb * 0.90` without subtracting 1.5GB.

## Goal

Replace static local model scoring and VRAM constants with a programmatic, future-proof local recommendation engine that uses live or cached metadata, GPU VRAM, Ollama installed/catalog state, inferred agent requirements, specialty matching, deterministic scoring, install decisions, hardware deficit warnings, README updates, and correctly sorted `fallback_models`.

## Non-Goals

- Do not build a broad remote Ollama registry crawler.
- Do not fabricate Ollama tags from OpenRouter IDs.
- Do not rely on live network in tests.
- Do not put local models in `routing`.
- Do not make OpenRouter or Ollama registry availability mandatory for CLI completion.
- Do not keep static per-model `MODEL_SCORES` or `BASE_VRAM` in the active local recommendation path.

## Invariants

- Persist only `model` and `fallback_models` to config; never persist `routing`.
- Canonical local refs are `local/<ollama-name>`.
- Each agent/category gets at most one chosen local fallback.
- Missing or unconfirmed local installs are not written to config by the apply pipeline.
- `completeAiRecommendations()` remains synchronous.
- Metadata acquisition happens before finalization, through injected/testable providers.
- Local fallbacks are deduped and ordered last after cloud fallbacks.

## Architecture

Add an isolated local recommendation pipeline instead of expanding `recommendation-finalizer.js` into a monolith.

Pipeline:

1. Discover installed/local candidates from existing GPU/Ollama discovery.
2. Enrich candidates with OpenRouter metadata and Ollama manifest layer sizes where available.
3. Infer each agent/category requirement from `AGENT_MODEL_REQUIREMENTS` and `CATEGORY_MODEL_REQUIREMENTS`.
4. Normalize each local candidate into an explainable candidate card.
5. Estimate VRAM using registry/catalog weight and KV cache formula.
6. Hard-filter by fit, context, and incompatible specialty.
7. Score and rank remaining candidates per entry.
8. Append the best local fallback, create `keep` or `install` decision, and emit deficit warnings when no local candidate fits.

## Requirement Inference

For each config entry:

1. Prefer `AGENT_MODEL_REQUIREMENTS[entry.name]` for agents.
2. Prefer `CATEGORY_MODEL_REQUIREMENTS[entry.name]` for categories.
3. Cross-reference fallback-chain models with OpenRouter metadata.
4. Set `minContext` to the maximum discovered `context_length`; default to `32000` when metadata is absent.
5. Infer specialty:
   - `reasoning`: `sisyphus`, `oracle`, `ultrabrain`, `unspecified-high`
   - `coding`: `hephaestus`, `prometheus`, `visual-engineering`
   - `vision`: `multimodal-looker`
   - `fast`: `explore`, `scout`, `quick`
   - `general`: everything else

## VRAM Formula

Use the user-specified budget and no legacy 1.5GB subtraction:

```text
budgetGb = gpu.vramGb * 0.90
fits = (weightGb + kvCacheGb) < budgetGb
```

Weights memory:

```text
weightGb = sum(ollamaManifest.layers[].size) / 1e9
```

If registry weight is unavailable, fall back to conservative local catalog or installed-size metadata. Reject candidates when neither weight nor parameters can be determined safely.

KV cache memory:

```text
kvCacheGb = 0.08 * (minContext / 1000) * (parametersB / 8) * gqaFactor * quantizationFactor
```

Defaults:

- `gqaFactor = 0.25` only for known modern GQA families such as Llama 3.1 and Qwen 2.5.
- Unknown GQA state uses `1.0`.
- `quantizationFactor = 0.25` only when local KV cache quantization is actually known/enabled.
- Unknown quantization uses `1.0`.

## Scoring

Hard-filter before scoring:

- must fit `gpu.vramGb * 0.90`
- must support required context
- must not be embedding-only or incompatible with the required specialty
- must have sufficient metadata to estimate memory

Score:

```text
score = (parametersB * 10) + openRouterPopularityIndex + roleMatchBonus
roleMatchBonus = 50 when candidate specialty matches entry specialty, else 0
```

When OpenRouter popularity is absent, use `0`. Installed state is a tie-breaker, not a hard override.

## Risk Register

| Risk | Impact | Mitigation |
| --- | --- | --- |
| OpenRouter schema changes | Scoring instability | Parse stable fields only; default popularity to `0`. |
| Ollama manifest unavailable | Missing weight estimates | Use conservative local fallback; reject unsafe unknowns. |
| VRAM formula remains approximate | Possible runtime OOM | Use 90 percent budget and explain limitations. |
| Metadata fetch slows CLI | Bad UX | Fetch once, dedupe, bound concurrency, short timeouts. |
| Finalizer grows complex | Maintenance risk | Keep logic in new pure module; finalizer orchestrates only. |
| Existing tests rely on static scores | Test churn | Assert behavior and computed outputs instead. |
| Cached panel has obsolete local | Invalid recommendation | Revalidate cached locals through context-aware fit map. |

## Success Criteria

- Local fallback selection is data-driven by metadata, requirements, VRAM estimate, specialty, fit, and score.
- Static local score/VRAM constants are not used for recommendations.
- Metadata failures do not fail the CLI.
- Every recommended local has a `keep` or `install` decision.
- Hardware deficits are clear and actionable.
- `fallback_models` are deduped and local-last.
- No locals are written to `routing`.
- Tests, lint, and smoke dry-run pass.
